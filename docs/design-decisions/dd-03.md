# 03: Use native Flask forms instead of Flask-WTF / WTForms

## Meta

Status
: **Work in progress** - Decided - Obsolete

Updated
: 16-Jun-2026

## Problem Statement

Our application needs forms for user registration, login, question creation and answer submission.

We need to decide whether these forms should be implemented with Flask-WTF / WTForms or with standard HTML forms and Flask's built-in request.form handling.

At the current stage of development, the routes for registration and login already exist, but the actual form processing logic has not been fully implemented yet. Therefore, the decision about form handling should be made before the authentication logic is added step by step.

The project already uses a simple Flask structure with explicit routes, SQLite and plain SQL statements. Therefore, the form handling should stay understandable and consistent with the rest of the application.
## Decision

We are currently planning to use standard HTML forms together with Flask's request.form mechanism instead of Flask-WTF / WTForms.

This means that form fields will be defined directly in the HTML templates and submitted through normal HTTP POST requests. The submitted values will then be read inside the route functions with request.form.

This approach is currently preferred because the expected form logic is simple and directly connected to the corresponding route. For example, registration will need to check whether required fields are filled, whether the email address uses the HWR domain and whether the email address already exists. Login will need to check whether the submitted email and password match a user in the database.

Using native Flask forms would also keep the connection between HTML templates and Python routes easy to understand. The name attributes in the HTML inputs would directly match the keys used in request.form.

This decision is still marked as work in progress because the final registration and login logic has not been implemented yet. Flask-WTF / WTForms was considered, but for the current implementation plan, native Flask form handling appears to fit better with the existing project structure.

*Decision was taken by:* github.com/David-hwr

## Regarded Options

### Option 1: Native Flask forms with request.form

With this option, the application would use normal HTML forms and process the submitted values directly in the corresponding Flask route.

Pros:

- No additional form dependency would be required.
- Form handling would stay close to the existing Flask route structure.
- The HTML input names would directly match the keys used in request.form.
- The approach would fit the existing plain SQL and SQLite architecture.
- Simple validation could be implemented directly in the route functions.
- The logic would be easy to trace for all team members.


Cons:

- Validation logic would be written manually.
- Similar validation checks might be repeated in multiple routes.
- Form structure and validation would not be centralized in separate form classes.

### Option 2: Flask-WTF / WTForm

With this option, forms would be implemented as Python classes and validated through WTForms.

Pros:

- Validation rules could be defined centrally in form classes.
- Flask-WTF provides built-in CSRF protection.
- Form objects could reduce repeated validation code.
- It could be useful for larger applications with many complex forms.

Cons:

- Additional dependency and learning effort would be required.
- The templates would need to be adapted to render WTForms fields correctly.
- CSRF protection would require correct configuration and hidden form fields.
- Mixing WTForms with existing plain HTML forms could lead to confusing errors.
- For the current implementation, the application is expected to need only a few simple forms whose values can be handled directly with request.form
