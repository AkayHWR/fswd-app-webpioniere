# 03: Use a dedicated JSON API route or JSON mode on HTML routes

## Meta

Status
: Work in progress - **Decided** - Obsolete

Updated
: 20-Jun-2026

## Problem Statement

Our application needs at least 1 “headless” API that delivers a JSON file. 
This is a mandatory requirement of the project.

We had to decide whether the app should expose JSON through a separate API
route or whether existing HTML routes should return JSON 
when the URL contains a query parameter such as `?json`.

## Decision

We decided to use a dedicated JSON API route:

In `app.py`, this is implemented with `@app.route('/api/questions')`. The route
reads questions from the database and returns them with Flask's `jsonify`
function.

This keeps the JSON interface separate from the HTML dashboard. The dashboard
route can focus on user-facing rendering, search, hashtag filters and template
variables, while the API route can focus on returning data as jsons.

*Decision will be taken by:* github.com/AkayHWR.

## Regarded Options

### Option 1: Dedicated API route with `jsonify`

With this option, the application has a separate endpoint for JSON data, for
example `/api/questions`.

Pros:

- The API is clearly separated from the HTML pages.
- The route name makes it obvious that the response is JSON.
- The dashboard route can stay focused on rendering `dashboard.html`.
- Tests and external clients can call a stable endpoint without relying on a
  special query parameter.
- The API can later be extended independently, for example with more fields,
  filters or additional endpoints.

Cons:

- The app has one additional route to maintain.
- The API endpoint has to be documented separately.

### Option 2: Return JSON from existing HTML routes with `?json`

With this option, existing routes would check whether a URL contains `?json`.
If the parameter exists, the route returns a Python data structure and lets
Flask convert it into JSON. Otherwise, the same route renders the normal Jinja
template.

For StudySwap, this would mean a route such as `/dashboard?json` could return
question data instead of rendering `dashboard.html`.

Pros:

- Very little extra routing code is needed.
- The same route can serve both the HTML page and the JSON representation.
- It follows the pattern from the course example where `?json` switches
  the response from HTML to JSON.

Cons:

- One route would have two different responsibilities: rendering HTML and
  serving JSON.
- The response type depends on a query parameter, which is less explicit than
  an `/api/...` path.
- The routes already handle a lot of different data, so mixing API behavior into it would make the route
  harder to read.
- Future changes to the different routes could accidentally affect the JSON response.
- It is less clear for external users which URLs are meant as API endpoints.
