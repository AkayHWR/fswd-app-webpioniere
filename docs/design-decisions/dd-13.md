# 13: Store passwords as salted hashes using Werkzeug

## Meta

Status
: Work in progress - **Decided** - Obsolete

Updated
: 19-Jul-2026

## Problem Statement

StudySwap requires users to create a password during registration and enter it again when logging in.

The application therefore needs to decide how passwords should be stored in the SQLite database. Storing passwords directly as plain text would make the implementation simple, but it would expose all user passwords if the database were accessed by an unauthorized person.

A normal hash would prevent the original password from being stored directly. However, identical passwords would still produce identical hashes if no additional protection were used. This would make the stored passwords more vulnerable to precomputed attacks.

The team therefore had to decide whether passwords should remain unhashed, be stored as simple hashes, be hashed with an individual salt or be protected with both a salt and an additional pepper.

## Decision

We decided to store passwords as salted hashes using Werkzeug's `generate_password_hash()` function.

During registration, the entered password is passed to `generate_password_hash()` before it is stored in the database. The database therefore contains only the generated password hash and not the original password.

During login, the submitted password is checked against the stored hash using Werkzeug's `check_password_hash()` function. The application does not decrypt the stored password. Instead, Werkzeug verifies whether the submitted password produces a matching result.

Werkzeug automatically creates and includes an individual salt in each generated password hash. This means that two users who choose the same password can still receive different stored hashes.

The existing database column remains named `password`, but it now stores the generated password hash instead of the plain-text password.

An additional pepper was not selected because it would require another secret value to be stored and managed separately from the database. This would increase the configuration and deployment effort without being necessary for the current project scope.

The decision has been implemented in the registration and login routes. The sample users are also inserted with generated password hashes so that the same login logic can be used for both sample accounts and newly registered users.

## Regarded Options

### Option 1: Store passwords without hashing

With this option, the application would store the entered password directly in the SQLite database.

Pros:

- The implementation would be very simple.
- Passwords could be compared directly during login.
- No additional hashing functions would be required.
- Sample users could be inserted directly through SQL statements.

Cons:

- All passwords would be readable in the database.
- An unauthorized person with database access could see every user password.
- Users may reuse the same passwords for other services.
- A database leak would immediately expose the original passwords.
- This option would not provide appropriate protection for user credentials.

### Option 2: Store passwords as simple hashes without a salt

With this option, the application would apply a basic hash function to the password before storing it.

Pros:

- The original password would no longer be stored directly.
- Passwords could not be read immediately from the database.
- The implementation would require only a small change to registration and login.

Cons:

- Identical passwords would create identical hashes.
- Attackers could compare the stored hashes with precomputed hash tables.
- Fast general-purpose hash functions are not designed specifically for password storage.
- The application would have to implement the comparison logic carefully.
- This option would provide less protection than salted password hashing.

### Option 3: Store passwords as salted hashes using Werkzeug

With this option, the application uses Werkzeug's `generate_password_hash()` and `check_password_hash()` functions.

Pros:

- The original password is not stored in the database.
- Werkzeug automatically generates an individual salt.
- Identical passwords can produce different stored hashes.
- The functions are designed specifically for secure password storage and verification.
- The implementation integrates directly with Flask.
- The application does not need to create or store salts separately.
- Registration and login remain easy to understand.

Cons:

- Existing plain-text passwords need to be replaced with generated hashes.
- Sample users cannot be inserted with normal plain-text SQL values.
- Passwords cannot be recovered if a user forgets them.
- Registration and login must consistently use the Werkzeug functions.

### Option 4: Store passwords as salted hashes with an additional pepper

With this option, the password would be combined with an additional secret value before the salted hash is generated.

The pepper would be stored separately from the database, for example in an environment variable.

Pros:

- An attacker would need both the password database and the separate pepper value.
- A leaked database alone would be less useful for password attacks.
- The pepper would add another layer of protection.

Cons:

- The pepper would need to be stored and managed securely.
- All application environments would need access to the same pepper.
- Losing or changing the pepper could prevent all users from logging in.
- The configuration and deployment process would become more complex.
- Werkzeug's built-in salted password hashing already provides suitable protection for this application.
