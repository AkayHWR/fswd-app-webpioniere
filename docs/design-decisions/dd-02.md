# 02: Use first name and last name instead of usernames

## Meta

Status
: Work in progress - **Decided** - Obsolete

Updated
: 15-Jun-2026

## Problem Statement

Our application needs to display the author of questions and answers and identify users in features such as the dashboard, question detail pages and the leaderboard.

We had to decide whether users should be able to create a separate username or whether the application should use their real first name and last name.

## Decision

We decided not to introduce a separate username field. Instead, the application stores and displays users through their first name and last name.

The user table therefore contains the columns:

first_name TEXT NOT NULL,
last_name TEXT NOT NULL.

We chose this approach because the application is designed as a university-related learning platform. In this context, using first name and last name makes it easier to recognize who created a question, wrote an answer or provided a solution.

Instead of introducing a separate username, the application can use the existing name information from the registration process wherever a user needs to be displayed. This keeps the user model simpler and avoids an additional registration field.

The exact way in which the full name is displayed can be handled during implementation. The important design decision is that the application does not need a separate username field.

*Decision was taken by:* github.com/David-hwr

## Regarded Options

### Option 1: First name and last name

With this option, the user registers with his real first name and last name.

Pros:

- The user model stays simple.
- No additional username field is required.
- Authors of questions and answers are easier to recognize.
- It fits the university context of the application.
- The leaderboard can display understandable real names.
- Registration stays straightforward.

Cons:

- Users have less anonymity.
- Users cannot choose a custom display name.
- Name changes would require editing first name or last name separatel.

### Option 2: Separate username

With this option, the user would choose an additional username during registration.

Pros:

- Users could choose their own public display name.
- Usernames can provide more privacy.
- Display names would stay independent from real names.

Cons:

- Registration would need one more input field.
- A separate username would make the registration and user table slightly more complex.
- For the current plant implementation, users only need to be displayed as authors of questions and answers. This can already be done with first name and last name.
