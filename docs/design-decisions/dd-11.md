# 11: Verify student HWR email addresses

## Meta

Status
: Work in progress - **Decided** - Obsolete

Updated
: 19-Jul-2026

## Problem Statement

StudySwap is intended for students of HWR Berlin. At the moment, the application only checks whether the entered email address ends with `@stud.hwr-berlin.de`.

This check prevents users from registering with email addresses from unrelated domains. However, it does not confirm whether the entered email address actually exists or whether the person registering has access to the corresponding mailbox.

For example, a user could enter an invented address such as `example@stud.hwr-berlin.de`. The current function would accept this address because it has the correct ending, even though the email account may not exist.

The current implementation is therefore sufficient for checking the domain, but it does not provide reliable proof that the user is an HWR student. We need to consider whether an additional verification step should be added and whether this would still fit the scope of the project.

## Decision

We decided to keep the current domain check for student HWR email addresses and not add an additional verification step.

One possible solution would be to send a verification link or a temporary verification code to the entered email address. The user would then have to open the link or enter the code before the account could be fully activated.

The existing domain check remains as the only verification step. It rejects email addresses that do not end with @stud.hwr-berlin.de.

An additional verification step was not selected because it was unclear which email service could be used, how the verification data should be stored and how much additional implementation effort would be required.

The team concluded that the verification process could not be implemented reliably within the remaining development time and the scope of the project. Therefore, the application continues to use only the existing domain check.

*Decision was taken by:* github.com/David-hwr

## Regarded Options

### Option 1: Keep the current domain check

With this option, the application would continue to check only whether the entered email address ends with `@stud.hwr-berlin.de`.

Pros:

- The current solution is already implemented.
- No additional service or dependency is required.
- Registration remains simple and quick.
- The development effort stays low.
- No changes to the database structure are necessary.

Cons:

- Invented email addresses can still be accepted.
- The application cannot confirm that the mailbox exists.
- The application cannot confirm that the user owns the entered email address.
- The restriction to HWR students can be bypassed by entering a false address with the correct ending.

### Option 2: Send a verification link or code by email

With this option, the application would send a verification link or temporary code to the entered student email address.

The account could remain inactive until the user confirms the address.

Pros:

- The application could confirm that the user has access to the entered mailbox.
- Invented email addresses would not result in fully activated accounts.
- The restriction to HWR students would become more reliable.
- Users would receive clear confirmation that their registration was successful.

Cons:

- It is not yet clear which email service could be used.
- Additional routes and database fields would probably be required.
- Verification links or codes would need to be created and checked securely.
- Emails could be delayed or classified as spam.
- The implementation could be too complex
