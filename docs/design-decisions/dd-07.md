# 06: Leaderboard Timeframe: All-Time Ranking vs. Time-Boxed (Semester) Windows

## Meta
Status : **Decided**
Updated : 20-Jun-2026

## Problem Statement
For the StudySwap leaderboard, we had to determine the timeframe over which user points are aggregated. Gamification theory suggests that an "All-Time" leaderboard can eventually demotivate new users because early adopters build insurmountable leads (the "rich get richer" problem). We had to decide whether to implement an absolute All-Time Ranking or a Time-Boxed system (e.g., resetting the ranking every semester).

## Decision
We decided to go with Option 1 (All-Time Ranking).
*Decision was taken by:* github.com/KarlTebbe

## Regarded options

### Option 1: All-Time Ranking (Lifetime Score)
With this option, the SQL query aggregates every single vote and solution a user has ever received since the creation of their account.

Pros:
- Technical Simplicity: The SQL query remains performant and straightforward without the need to filter by complex timestamp logic (`created_at`).
- Hall of Fame Effect: It creates a strong sense of long-term prestige for highly active core users, effectively building a "Hall of Fame" for the university's best helpers.
- UI Simplicity: No need for additional frontend elements (like dropdowns or tabs) to let users switch between "This Week", "This Semester", or "All Time".

Cons:
- Newcomer Demotivation: After a few semesters, the top ranks will be cemented. Freshmen joining the platform will find it mathematically nearly impossible to reach the top 10, potentially discouraging them from participating.

### Option 2: Time-Boxed / Rolling Ranking (e.g., Current Semester)
With this option, the system only counts votes cast within a specific timeframe (e.g., since the official start of the semester).

Pros:
- High Continuous Engagement: The competition resets regularly, giving every new cohort of students a fair chance to climb to the top of the leaderboard.
- Reflects Current Activity: The leaderboard highlights users who are currently active and helpful, rather than students who might have already graduated.

Cons:
- Backend Complexity: Requires complex date-math in SQL.
- Loss of Long-Term Investment: Users might feel their hard work is "erased" every semester, leading to a drop in motivation to write high-effort, comprehensive answers.

## Rationale
For the initial MVP (Minimum Viable Product) launch of StudySwap, establishing a core user base and keeping the architectural complexity low is the main priority. An All-Time ranking is the most robust starting point to prove the core platform loop works. Advanced gamification mechanics, such as seasonal resets or semester-based leaderboards, are deliberately deferred to future development cycles once the platform has reached a critical mass of active users.
