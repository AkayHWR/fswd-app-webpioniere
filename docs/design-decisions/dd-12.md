# 12: Support image uploads for questions and answers

## Meta

Status
: **Work in progress** - Decided - Obsolete

Updated
: 18-Jul-2026

## Problem Statement

StudySwap should support images because users may want to add screenshots, diagrams, lecture notes or photos to questions and answers. The team needs to decide how images should be uploaded, stored, validated and displayed.

This decision affects the database model, Flask routes, forms, templates, static files and security checks.

## Decision

*Decision will be taken by:* github.com/AkayHWR.

## Regarded Options

### Option 1: Store image files in the filesystem and metadata in SQLite

With this option, uploaded images are saved as files and the database stores references to them.

Pros:

- SQLite stays small and focused on structured data.
- Images can be served as static files or through a controlled Flask route.
- Metadata can be queried together with questions and answers.
- This fits the current project size and local development setup.

Cons:

- The upload folder must be handled carefully and should not accidentally contain committed user uploads.
- Deleting or archiving content must also consider related image files.
- The team needs validation for file type, size and filenames.

### Option 2: Store images directly as BLOBs in SQLite

With this option, the image binary data is stored inside the database.

Pros:

- Each image stays together with the database record.
- Backup can be simpler because fewer filesystem paths are involved.
- There is no separate upload folder to manage.

Cons:

- The SQLite database can become large quickly.
- Serving images from database BLOBs adds extra route logic.
- It is less convenient for normal static file handling.
- It may make local debugging and inspection harder.

### Option 3: Use external object storage

With this option, images are uploaded to a service outside the application, such as cloud object storage.

Pros:

- This can scale better for larger production systems.
- The application server does not need to store uploaded files locally.
- CDN or public/private file access rules could be added later.

Cons:

- It adds an external dependency and configuration.
- It is too complex for the current course project scope.
- It may require credentials and deployment setup that the team does not currently need.
