# 00: Use plain SQL statements or SQLAlchemy

## Problem Statement

Our application needs persistent data for users, questions, answers and votes. The database layer must support basic CRUD operations, joins and foreign key relationships while staying understandable for all team members.

We have to decide whether to interact with the SQLite database through explicit SQL statements or through an object-relational mapper such as SQLAlchemy.

## Regarded Options

### Option 1: Plain SQL with SQLite

With this option, the application uses SQL statements directly.

Pros:

- The executed queries are visible and easy to trace.
- No additional dependency besides Flask and Python's built-in SQLite support is required.
- The SQL schema in `sql/create_tables.sql` directly matches the database structure.
- Primary keys, foreign keys and joins are explicit and easier to explain in the documentation.
- The approach fits the small size of the application.

Cons:

- More SQL has to be written manually.
- Changing to another database system later would require more code changes.
- Repeated query patterns are not abstracted automatically.

### Option 2: SQLAlchemy / Flask-SQLAlchemy

With this option, the application would define Python model classes and let SQLAlchemy map them to database tables.

Pros:

- Database access can be written with Python objects instead of SQL strings.
- Relationships between tables can be accessed more conveniently through model attributes.
- Switching from SQLite to another database system such as PostgreSQL would be easier.
- Larger applications can benefit from a clearer model layer.

Cons:

- SQLAlchemy adds another dependency and another concept the team has to learn.
- The actual SQL becomes less visible.
- The project would need model classes, sessions and ORM-specific patterns.
- For the current project size, the abstraction would add complexity without a strong practical benefit.
